What changed and why
=====================

The extension had bad behavior when optional dependencies weren't fully installed.
Specifically the wake word listener (the "hey Friday" feature) and the comms panel
passthrough had issues.


Problem 1 — Wake daemon flashing then crashing
-----------------------------------------------
When python3 was installed but the actual wake word libraries (openwakeword, pyaudio)
were missing, the daemon would spawn, briefly show "DAEMON ON" in the status bar,
then immediately crash. This was confusing — it looked like the feature was broken
rather than simply unavailable.

Fix: Before spawning the daemon process, verify that the required Python modules can
actually be imported. If they can't, skip spawning entirely and log a message. No
flash, no crash, no confusion.


Problem 2 — Status bar showing irrelevant daemon state
-------------------------------------------------------
The status bar showed "DAEMON OFF" even on machines where the wake word feature
couldn't possibly work (missing deps). This is noise — if you can't run the feature,
you shouldn't see its status at all.

Fix: Only show daemon status (ON or OFF) when the system actually has the deps to
run it. Otherwise, that section of the status bar is simply absent.


Problem 3 — Status bar not updating when daemon exits
------------------------------------------------------
When the daemon process died, the status bar kept showing "DAEMON ON" until the next
UI refresh event happened to trigger an update.

Fix: When the daemon process exits, immediately refresh the status bar. To support
this, the UI context is now cached so status updates can happen without waiting for
an event that carries a UI reference.


Problem 4 — Comms panel message ordering
------------------------------------------
Assistant text was being forwarded to the comms panel at the end of the entire turn,
which meant it arrived after tool calls had already executed. This caused messages
to appear out of order — context text showing up after the main response.

Fix: Forward assistant text as soon as each message finishes streaming, before tool
calls execute. This gives proper ordering: context first, then the communicate
message on top.


Problem 5 — Listen command giving wrong error
-----------------------------------------------
The /friday listen command told users they needed "python3" when the actual
requirement is python3 plus openwakeword plus pyaudio. Misleading if you have
python3 but not the libraries.

Fix: Updated the error message to name all three requirements, and gated the
command on the full dep check instead of just the python3 binary.


For a different agent patching this in the future
==================================================

The core concept: there are two levels of dependency checking.

Level 1 is binary presence — does the command exist on PATH? This is cheap and
happens at startup. Used for piper, sox/play, python3.

Level 2 is functional verification — can the tool actually do its job? For the
wake word feature, this means python3 can successfully import openwakeword and
pyaudio. This is slightly more expensive (spawns a quick python process) but
only runs once at startup.

The wake word feature should only be considered "available" when BOTH levels pass.
Everything that gates on wake word availability — daemon spawning, status bar
display, keyboard shortcuts, slash commands — should use the Level 2 check.

The status bar helper should be callable without arguments for reactive updates
(daemon exit, settings change). Cache whatever UI context it needs from the last
event-driven call so it can refresh itself independently.

The comms panel passthrough should fire per-message, not per-turn. The relevant
event is when a single assistant message finishes streaming, not when the entire
turn (which may include multiple messages and tool calls) completes.
